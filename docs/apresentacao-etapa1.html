O que é a Etapa 01

O processador base entregue pelos professores já suportava apenas algumas instruções fundamentais: ADD, SUB, AND, OR, SLT, LW, SW e BEQ. A Etapa 01 consiste em expandir a ALU e a unidade de controle para suportar mais 12 instruções — 5 R-type e 7 I-type aritmético/lógico — sem alterar o pipeline em si.

O que foi modificado e por quê

pl_alu.sv — Novas operações na ALU

A ALU é um case em Operation[3:0]. Bastou adicionar 5 novos casos:

// Já existia: 4'd01: ALUResult = $signed(SrcA) + $signed(SrcB); // ADD 4'd04: ALUResult = SrcA | SrcB; // OR 4'd05: ALUResult = SrcA & SrcB; // AND 4'd11: ALUResult = 32'($signed(SrcA) < $signed(SrcB)); // SLT // Adicionado na Etapa 01: 4'd03: ALUResult = SrcA ^ SrcB; // XOR ← novo 4'd06: ALUResult = SrcA << SrcB[4:0]; // SLL ← novo 4'd07: ALUResult = SrcA >> SrcB[4:0]; // SRL ← novo 4'd08: ALUResult = $signed(SrcA) >>> SrcB[4:0]; // SRA ← novo 4'd12: ALUResult = 32'($unsigned(SrcA) < $unsigned(SrcB)); // SLTU ← novo

pl_alu_ctrl.sv — Decodificação de funct3/funct7

Traduz ALUOp + funct3 + funct7[5] para o código de operação da ALU. Dois novos blocos foram adicionados:

// R-type (ALUOp = 2'b10) — já existia ADD/SUB, AND, OR, SLT: 2'b10: case (Funct3) 3'h0: Operation = Funct7[5] ? 4'd02 : 4'd01; // SUB / ADD 3'h1: Operation = 4'd06; // SLL ← novo 3'h3: Operation = 4'd12; // SLTU ← novo 3'h4: Operation = 4'd03; // XOR ← novo 3'h5: Operation = Funct7[5] ? 4'd08 : 4'd07; // SRA/SRL ← novo 3'h6: Operation = 4'd04; // OR 3'h7: Operation = 4'd05; // AND endcase // I-type aritmético (ALUOp = 2'b11) — bloco inteiramente novo: 2'b11: case (Funct3) 3'h0: Operation = 4'd01; // ADDI 3'h4: Operation = 4'd03; // XORI 3'h6: Operation = 4'd04; // ORI 3'h7: Operation = 4'd05; // ANDI 3'h1: Operation = 4'd06; // SLLI 3'h2: Operation = 4'd11; // SLTI 3'h5: Operation = Funct7[5] ? 4'd08 : 4'd07; // SRAI/SRLI endcase

pl_control.sv — Novo opcode I-type aritmético

O controlador principal só enxerga o opcode (7 bits). O opcode das instruções I-type aritméticas (ADDI, ORI, etc.) é 7'b0010011, diferente do LOAD (7'b0000011). Foi adicionado um novo caso:

localparam I_TYPE = 7'b0010011; // ← novo I_TYPE: begin ALUSrcB = 1'b1; // SrcB da ALU vem do imediato, não de rs2 RegWrite = 1'b1; // escreve resultado em rd ALUOp = 2'b11; // sinaliza ao alu_ctrl que é I-type aritmético end

pl_sign_ext.sv — Extensão de sinal do imediato

As instruções I-type têm um imediato de 12 bits em Instr[31:20]. O extensor de sinal já tratava LOAD com o mesmo encoding — a solução foi compartilhar o mesmo caso:

localparam I_TYPE = 7'b0010011; localparam LOAD = 7'b0000011; // Ambos usam o mesmo encoding de imediato: LOAD, I_TYPE: ImmExt = {{20{Instr[31]}}, Instr[31:20]}; // ↑ extensão de sinal de 12 para 32 bits

Decisões de projeto

Reutilização de operações da ALU

As 7 instruções I-type não precisam de novas entradas na ALU. ADDI reutiliza ADD, ANDI reutiliza AND, etc. Só o alu_ctrl precisa mapear o novo funct3 para o código já existente. Isso manteve a ALU simples.

ALUOp 2'b11 para I-type

O código base usava ALUOp=10 para R-type e 01 para branch. Precisávamos de um código novo para I-type aritmético, já que LOAD também usa imediato mas quer forçar ADD (ALUOp=00). Escolhemos 2'b11 — o único valor livre.

SRA e SRAI: bit funct7[5]

SRL e SRA têm o mesmo funct3=101. O ISA usa o bit 30 da instrução (= funct7[5]) para diferenciá-los: 0 = lógico, 1 = aritmético. O mesmo bit diferencia SRLI de SRAI no I-type — o encoding é idêntico.

Shifts usam só SrcB[4:0]

Em RISC-V, o shift amount é sempre os 5 bits menos significativos de rs2 (ou do imediato), pois o máximo útil para 32 bits é 31. A ALU usa SrcB[4:0] — os outros 27 bits são ignorados por definição do ISA.

sign_ext compartilha LOAD e I_TYPE

LOAD e I-type aritmético têm o mesmo encoding de imediato: Instr[31:20] com extensão de sinal. Em vez de duplicar código, o pl_sign_ext agrupa os dois opcodes no mesmo case do SystemVerilog.

NOPs antes dos stores na simulação

Com pipeline de 5 estágios, uma instrução recém-calculada demora até 4 ciclos para estar disponível em WB. Para que o testbench imprima o estado final correto, inserimos 4 NOPs entre os cálculos e os SW, garantindo que todos os registradores já foram escritos antes de ir para a memória.

Perguntas possíveis na apresentação
?
Qual a diferença entre SRL e SRA?
SRL (Shift Right Logical) preenche os bits mais significativos com zero — ignora o sinal. SRA (Shift Right Arithmetic) replica o bit de sinal (bit 31) nos bits que entram. Por isso SRA -8, 3 dá -1 (todos os bits ficam 1), enquanto SRL -8, 3 daria um número positivo grande. No código, usamos >>> do SystemVerilog com o cast $signed para forçar o comportamento aritmético.
?
Como o processador sabe se é SRA ou SRL se ambos têm funct3=101?
O ISA RISC-V usa o bit 30 da instrução — que coincide com funct7[5] — como discriminador. Se funct7[5]=1, é aritmético; se 0, é lógico. O pl_alu_ctrl.sv verifica esse bit: Operation = Funct7[5] ? 4'd08 : 4'd07. O mesmo vale para SRAI vs SRLI no I-type.
?
Por que ADDI não precisou de nova operação na ALU?
Porque ADDI rd, rs1, imm faz exatamente a mesma operação que ADD — soma dois valores de 32 bits. A única diferença é que um dos operandos vem do imediato e não de um registrador. Quem cuida disso é o sinal ALUSrcB=1 no controlador, que direciona o mux de entrada da ALU para o imediato. A ALU em si recebe dois valores e soma — não sabe nem se importa de onde vieram.
?
Qual a diferença entre SLT e SLTU?
SLT compara os operandos com sinal (complemento de dois): -1 < 1 é verdadeiro. SLTU compara sem sinal: o mesmo padrão de bits de -1 (0xFFFFFFFF) é tratado como 4294967295, que é maior que qualquer número de 32 bits positivo. No código, usamos os casts $signed e $unsigned do SystemVerilog para forçar o tipo antes da comparação.
?
Por que foi necessário criar o ALUOp 2'b11?
O controlador da ALU (pl_alu_ctrl) precisa saber em qual contexto vai decodificar funct3. Com só 2 bits de ALUOp, os valores existentes eram: 00 = Load/Store (forçar ADD), 01 = Branch (forçar SUB), 10 = R-type. Precisávamos de um quarto caso para I-type aritmético — 2'b11 foi o valor natural disponível. Se tivéssemos usado 10 (mesmo que R-type), o decodificador não saberia se deve ignorar funct7 ou não.
?
O que é o imediato nas instruções I-type e como é estendido?
As instruções I-type têm um campo de 12 bits codificado em Instr[31:20]. Para operar com 32 bits, esse valor precisa ser estendido para 32 bits preservando o sinal: os 20 bits mais significativos são preenchidos com cópias do bit 31 (bit de sinal). Isso é feito em pl_sign_ext.sv: ImmExt = {{20{Instr[31]}}, Instr[31:20]}. Assim, imediatos negativos como -8 (111111111000 em 12 bits) ficam corretamente representados em 32 bits.
?
Como foi feita a divisão do trabalho entre os integrantes?
3 instruções por pessoa. O critério foi agrupar instruções com lógica similar para minimizar conflitos no git e facilitar o entendimento:

Caio — XOR, ADDI, ANDI  |  Lucas — SLL, ORI, SLTI  |  João — SRL, SRA, SLLI  |  Thales — SLTU, SRLI, SRAI

Os arquivos compartilhados (pl_control.sv e pl_sign_ext.sv) foram alterados por uma única pessoa antes dos demais — assim ninguém teve conflito de merge nesses dois arquivos.
?
A etapa 01 afeta o pipeline? Gera hazards novos?
Não. Todas as 12 instruções são resolvidas no estágio EX — nenhuma acessa memória nem altera o PC. Os hazards existentes (forwarding para RAW e load-use stall) já cobriam esse caso: se uma instrução R-type ou I-type produz um resultado que outra instrução lê em seguida, o forwarding encaminha o dado sem stall. O único cuidado foi nos NOPs inseridos na simulação para dar tempo ao testbench de capturar o estado final.
?
Por que usar $signed e $unsigned em SystemVerilog?
Por padrão, SystemVerilog trata sinais de logic como sem sinal. Para que operadores como < e >>> se comportem corretamente com negativos, precisamos explicitamente informar o tipo com $signed(). Sem esse cast, SRA -8, 3 produziria o resultado de SRL (shift lógico) — o simulador simplesmente não replica o bit de sinal.
